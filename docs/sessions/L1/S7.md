## 0) PEP 8: Code Style Warm-Up (~4 min)

You already learned the core PEP 8 habits in [Session 5](S5.md). This warm-up is only a quick recall before debugging, not a second full style lesson.

Keep these three habits active while you read tracebacks and refactor buggy code:

- use clear `snake_case` names so variables are easy to follow in error messages
- keep spaces around operators and after commas so expressions are readable at a glance
- write comments that explain **why**, not just **what**
- keep `#` comment lines to about **72 characters** when you can — split long `# Why:` notes onto a second line

### Indentation focus for debugging

This session adds one extra style detail because indentation errors often look like debugging problems to beginners.

- Use **4 spaces** per indent level.
- When a long function call wraps, align it cleanly on the next line.

```python
result = some_function(
  argument_one,
  argument_two,
  argument_three,
)
```

**Why it matters now:** Python uses indentation to define blocks, so messy layout can become `IndentationError` or hide the real branch that is running.

If you want the full five-rule refresher, go back to the PEP 8 review in [Session 5](S5.md) and then return here for debugging practice.

## 1) Syntax vs Runtime Errors

| Error Type | When it appears | Typical examples |
| --- | --- | --- |
| Syntax error | Before program runs | Missing `:`, indentation mistake, unmatched quote |
| Runtime error | During execution | `ZeroDivisionError`, `TypeError`, `IndexError`, `NameError` |

**Reasoning:** If Python cannot parse your code, execution never starts. If parsing succeeds, logic/data issues can still fail at runtime.

## 2) How to Read a Traceback

Use this sequence every time:

1. Start at the **bottom** (actual error type + message).
2. Find file + line number where it failed.
3. Inspect that line and nearby lines.
4. Trace backward to the input/state that produced it.

Common errors for Session 7:

- `NameError` → variable not defined
- `TypeError` → incompatible operation for type
- `ValueError` → invalid value for conversion
- `IndexError` → index outside list bounds
- `KeyError` → dictionary key missing
- `ZeroDivisionError` → divisor is zero

## 3) Common Beginner Mistakes to Watch

- Missing colons in `if/for/while` headers
- Misaligned indentation blocks
- Using `=` when comparison needs `==`
- Comparing string input to integer values without conversion
- Off-by-one assumptions in `range()`
- Forgetting loop updates in `while` loops
- **Compound conditions that “look right” but evaluate wrong** — e.g. mixing `and` / `or` / `not` without parentheses. Python applies **`not` before `and` before `or`** unless you override with `()`. Symptom: loop never runs, runs forever, or branch flips the opposite way — **no traceback**, just wrong behavior.

**Debug tactic:** print the **pieces** of the condition before the `while` / `if`:

```python
ready = count < limit
allowed = not cancelled
print("DEBUG:", ready, allowed, ready and allowed)
while ready and allowed:
    ...
```

For a guided refresher on evaluation order and a tiny compound `while`, run (from repo root): `python src/L1/S6/08_boolean_logic_precedence.py` (see [Session 6](S6.md)).

## 4) Print-Debugging Workflow

When behavior is wrong but script still runs:

1. Print inputs before logic starts.
2. Print values at each decision point.
3. Print loop counters and intermediate totals.
4. Confirm final output against expectation.
5. Remove temporary debug prints after fix.

Use a prefix like `DEBUG:` so cleanup is fast.

## 7) Troubleshooting Quick Guide

- `SyntaxError` / `IndentationError` → fix structure first
- `NameError` → verify exact variable names
- `TypeError` / `ValueError` → validate and convert input early
- `IndexError` / `KeyError` → check existence before access
- `ZeroDivisionError` → guard divisor before dividing
- **Wrong behavior, no exception** → print sub-expressions in `if` / `while` headers; confirm **`not` / `and` / `or`** order and parentheses (Session 6 optional `08_boolean_logic_precedence.py`)

## 📚 Appendix: Concepts Revisited from Session 1 (formatted output)

### print() with sep and end

`print()` accepts any number of values. By default it joins them with a space (`sep=" "`) and adds a newline at the end (`end="\n"`). Changing these two parameters is enough to produce clean debug output on a single line.

```python
name = "Alice"
score = 95

# Default: space separator, newline at end
print(name, score)            # Alice 95

# Custom separator
print(name, score, sep=" | ") # Alice | 95

# Suppress newline — stays on same line
print("Score:", end=" ")
print(score)                  # Score: 95
```

Common mistake: forgetting the comma — `print(name score)` is a `SyntaxError`.

See `src/L1/S7/06_print_sep_end.py` for a full walkthrough.

### Escape sequences

A backslash inside a string starts an **escape sequence** — it changes how the next character prints rather than appearing literally.

| Sequence | Meaning | Typical debug use |
| --- | --- | --- |
| `\n` | New line | Separate sections in multi-line output |
| `\t` | Horizontal tab | Indent / align columns |
| `\b` | Backspace | Terminal cursor effect (rarely needed) |
| `\'` | Literal apostrophe inside single-quoted string | `'I\'m ready'` |
| `\"` | Literal quote inside double-quoted string | `"She said \"hello\""` |

```python
print("Label:\tValue")          # Label:  Value
print("Step 1\nStep 2\nStep 3") # three lines
print('It\'s working')          # It's working
```
